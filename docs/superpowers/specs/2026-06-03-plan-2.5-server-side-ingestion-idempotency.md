# Plan 2.5: Server-side ingestion idempotency

**Status:** spec (hardened by adversarial review)
**Repo:** `epigraph` (public) · **Branch base:** `docs/commit-ingestion-spec`
**Depends on / unblocks:** the PR-hierarchical commit ingester — Plan #2
(`docs/superpowers/specs/2026-06-02-commit-ingestion-automation-design.md` +
`docs/superpowers/plans/2026-06-02-pr-hierarchical-ingestion-core.md`). This plan is a
**prerequisite for Plan #3** (cross-repo GitHub Actions rollout).

---

## 0. Hardening summary (from adversarial review)

A first draft proposed routing `submit/packet` through the existing `create_or_get`
helper and making agent registration idempotent. Three independent reviewers (blast-radius,
completeness, codebase-fit) verified the core approach is correct **but unsound as first
written**. This spec folds in the **five blocking fixes** they found — each is a real
correctness defect, not polish:

- **B1 — Properties data-loss on the NORMAL path.** `create_strict`/`create_or_get` do
  **not** write the `properties` column; the bare INSERT being replaced does. Without porting
  the `properties` write, *every* `submit/packet` caller silently loses `properties` —
  including commit metadata and telemetry's `properties->>'event'` marker (which would shove
  telemetry claims back into the embedding queue). See §2.4.
- **B2 — Phantom `evidence_ids`/`trace_id` on the dedup path.** A `(ClaimId, bool)` return is
  insufficient: the response still carries the pre-generated `trace_id`/`evidence_ids` that
  were never inserted on dedup, and the CLI then POSTs embeddings to non-existent evidence
  (404s). The dedup path must return **empty `evidence_ids`** and the **existing** claim's
  `trace_id`. See §2.3.
- **B3 — Dead-node trap.** `uq_claims_content_hash_agent` has **no `is_current` predicate**,
  and `find_by_content_hash_and_agent` neither filters nor returns `is_current`. Once a
  `(content_hash, agent_id)` is superseded / `mark_duplicate`'d (row kept, `is_current=false`,
  embedding NULL), `create_or_get` is permanently stuck returning the tombstone and the
  ingester would link edges to a dead node. See §2.5.
- **B4 — `claim_id` rebind / `was_duplicate` threading.** The dedup id is trapped inside the
  `cfg(feature="db")` block; the response, `ClaimSubmitted` event, in-memory propagation, and
  idempotency-cache write all use the OUTER pre-generated UUID and `was_duplicate:false`. The
  outer `claim_id` must be rebound to the returned id and `was_duplicate` threaded out;
  the `cfg(not(db))` build must be handled. See §2.3.
- **B5 — Propagation orchestrator on the dedup path.** The propagation block
  (`submit.rs:~1395-1469`, NOT `cfg(db)`-gated) calls `register_claim` / `propagate_from` on
  the phantom id every submit. It must be **skipped on `was_duplicate==true`**. See §2.2.

Plus accepted improvements: edge-skip reframed as graph-hygiene not a 500-fix (migration 018
dropped the triple-unique index); `claim.created` event now fires on the create path (owned in
§2.4); OAuth-skip guard is moot for the unauthenticated CLI but matters for token callers (§3.2);
prod constraint-liveness check before relying on `DuplicateKey` (§7); stale "migration 107"
comment in the template clarified (§4).

## 1. Motivation

Plan #2 runs the ingester **once per merged PR, in a fresh GitHub Actions process**. Its
design (§8) requires *server-side* find-or-create for the repo node, author agents, and
orchestrator agents. Against the current server two operations 500 (or `exit(1)` the CLI) on
run 2+.

### 1.1 Claim insert is not idempotent across processes
`submit_packet → persist_packet` (`crates/epigraph-api/src/routes/submit.rs`, `INSERT INTO
claims (...)` at ~line 1029) is a **bare INSERT, no `ON CONFLICT`, no pre-check**. `claims`
carries `uq_claims_content_hash_agent UNIQUE (content_hash, agent_id)`
(`migrations/013_code_review_hardening.sql:25`), **live in this repo** (the "migration 107
dropped it" comment at `claims.rs:447` is *internal*-repo history; no such migration exists
here — migrations stop at 049). A cross-run resubmit of the same `(content_hash, agent_id)` →
**500**. The only dedup is the per-process LRU `state.idempotency_store`
(`crates/epigraph-api/src/state.rs`; read at `submit.rs:1322`, written at `submit.rs:1671`) —
it does not survive across processes.

### 1.2 Agent registration is not idempotent on public key
`AgentRepository::create` (`crates/epigraph-db/src/repos/agent.rs:79`) maps the
`agents_public_key_unique` violation (`migrations/001_initial_schema.sql:1583`) to
`DbError::DuplicateKey` → **400** (`errors.rs:159`). `AgentRepository::get_by_public_key`
**exists** (`agent.rs:194`) but is **not on any HTTP route** (`mod.rs:155-157,503-506` wire
only POST create, GET by id, list-by-label). The CLI's `register_agent`
(`crates/epigraph-cli/src/bin/ingest_git.rs:919`) treats non-2xx as fatal → `exit(1)`
(lines 1618-1621, 1633-1636). So a deterministic-keyed agent can neither be re-registered nor
recovered on run 2+.

## 2. Fix A — `submit/packet` claim find-or-create on `(content_hash, agent_id)`

### 2.1 What to reuse (already exists)
- `ClaimRepository::find_by_content_hash_and_agent(conn, content_hash, agent_id)` — `claim.rs:1764`.
- `ClaimRepository::create_strict(conn, claim)` — `claim.rs:1817` (unconditional insert;
  surfaces `DuplicateKey`; emits transactional `claim.created`).
- `ClaimRepository::create_or_get(conn, claim) -> (Claim, bool /*was_created*/)` — `claim.rs:1912`
  (find by `(content_hash, agent_id)` → existing; else `create_strict`; catches the race
  `DuplicateKey`, re-finds; takes `&mut PgConnection`).

`POST /api/v1/claims` (`create_claim`, `claims.rs:444`) is the **template** — it calls
`create_or_get(&mut tx, &claim)` and branches on `was_created`. **Mirror it faithfully,
including the parts §2.4/§2.5 add** (the first draft mirrored only the call, not the
`properties` UPDATE — that omission is blocking-fix B1).

### 2.2 The change to `persist_packet` (`submit.rs:974`)
Replace the bare `INSERT INTO claims` (submit.rs:1029) with `create_or_get(&mut *tx, &claim)`
(tx already open at submit.rs:1006). **Mandatory short-circuit when `was_created == false`:**

1. **Skip the evidence insert** — `evidence_content_hash_claim_unique UNIQUE (content_hash,
   claim_id)` (`001:1821`) would 500 on identical re-submit (this one IS constraint-driven).
2. **Skip the reasoning-trace insert and the `trace_id` UPDATE** (submit.rs ~1068-1109) —
   the UPDATE would mutate another run's canonical noun-claim, forbidden per `claims.rs:484-487`.
3. **Skip the structural-edge inserts** — `AUTHORED`, `HAS_TRACE`, **and `TRACES`**
   (submit.rs:1112-1158). **Reframed (improvement):** migration `018_drop_edges_triple_unique_constraint.sql`
   dropped the triple-unique index and these are bare INSERTs, so re-inserting would NOT 500 —
   it would create **duplicate edge rows**. Skipping is **graph-hygiene** (matches `create_claim`,
   which does not re-edge on dedup), not a 500-fix.
4. **(B5) Skip the propagation-orchestrator block** (`submit.rs:~1395-1469`, not `cfg(db)`-gated):
   on the dedup path do **not** call `orchestrator.register_claim(...)` / `propagator.propagate_from(...)`
   — they would run on an already-existing node under the (now-corrected, §2.3) id. Re-registering
   an existing node is wrong and wasteful. (Severity: in-memory/per-process, truth-persistence is
   commented out at submit.rs:1452-1457 — a wiring defect, not live corruption — but a literal
   implementer of the first draft leaves a phantom-id path, so this skip is mandatory.)

### 2.3 Signature & data-flow change (B2 + B4)
`persist_packet` currently returns `Result<(), …>` and the caller pre-generates
`claim_id = Uuid::new_v4()` (submit.rs:1354), used unconditionally for the response, the
`ClaimSubmitted` event, the in-memory `Claim`, propagation, the embedding UPDATE, and the
idempotency-cache write — **none of which are `cfg(db)`-gated**, while `persist_packet` is
called *inside* `#[cfg(feature="db")]` (submit.rs:1376).

Change `persist_packet` to return the real ids + dedup flag — e.g.
`Result<PersistOutcome, …>` where
`struct PersistOutcome { claim_id: ClaimId, trace_id: Option<TraceId>, evidence_ids: Vec<EvidenceId>, was_duplicate: bool }`.

- **On create:** `claim_id` = the inserted id (== pre-generated, since `create_strict` inserts
  `claim.id`), `trace_id`/`evidence_ids` = the freshly inserted ones, `was_duplicate=false`.
- **On dedup (B2):** `claim_id` = the **existing** row's id; `evidence_ids` = **empty**
  (the CLI's evidence-embedding loop, ingest_git.rs:~1429, iterates the response
  `evidence_ids`, so empty ⇒ no 404s and no re-embed of already-embedded evidence);
  `trace_id` = the **existing** claim's `trace_id` (fetch via the existing `get_by_id` on the
  dedup branch — see §2.5, which already does a `get_by_id` for the `is_current` check, so
  reuse that row; no new query macro). `was_duplicate=true`.

**Caller rewiring (B4) — mandatory:** rebind the OUTER `claim_id` to `outcome.claim_id`;
set `SubmitPacketResponse.was_duplicate = outcome.was_duplicate` (field exists, submit.rs:332);
use `outcome.claim_id` in the `ClaimSubmitted` event (submit.rs:1664), the idempotency-cache
write (submit.rs:1671+), the embedding step (§5), and skip propagation per §2.2 (B5). Return
**201** on both paths (matches the existing `idempotency_store` short-circuit, submit.rs:1326-1335).
Handle the **`cfg(not(db))`** build: that arm has no DB, so it keeps the pre-generated id and
`was_duplicate=false` (document that idempotency is a db-feature guarantee).

### 2.4 Properties preservation + `claim.created` event (B1, owned behavior change)
**B1 (blocking):** the replaced bare INSERT writes the `properties` JSONB column (submit.rs:1039);
`create_strict` writes **only** id/content/content_hash/truth_value/agent_id/trace_id/timestamps —
**no `properties`**. The template `create_claim` compensates with a separate
`UPDATE claims SET properties = … WHERE id = …` guarded on `was_created` (`claims.rs:488-507`).
**Port that UPDATE** into `persist_packet` on the **create path** (`was_created==true`), before
commit. Without it, every properties-bearing submit/packet caller silently loses `properties` —
verified-affected: `ingest_git` `build_packet` sets `properties` (ingest_git.rs:1111-1125), and
epiclaw-host telemetry relies on `properties->>'event'` (CLAUDE.md telemetry exception +
`find_claims_needing_embeddings`, claim.rs:2209-2210) — dropping it would re-queue telemetry for
embedding. Add a test (§10.8) asserting `properties` round-trips on the create path.

**Owned behavior change:** `create_strict` fires a transactional `claim.created` event
(claim.rs:1861) that the current bare INSERT does **not**. After this fix the create path writes a
`claim.created` row it never wrote before (the dedup path does not, since `create_strict` isn't
called). This is desirable (parity with MCP `create_claim_idempotent`) but must be called out so
events-table consumers aren't surprised.

### 2.5 Dead-node guard (B3, blocking)
`uq_claims_content_hash_agent` has **no `is_current` predicate**; `mark_duplicate` (claim.rs:~2076)
and `supersede` keep the row, set `is_current=false`, and NULL the embedding; and
`find_by_content_hash_and_agent` (claim.rs:1764) has **no `is_current` filter** and its SELECT
does not return `is_current`. So if a `(content_hash, agent_id)` has ever been superseded/marked-
duplicate, `create_or_get` returns the **tombstone** forever (the constraint forbids inserting a
live replacement), and the ingester would link `decomposes_to`/`RESOLVED_BY` edges to a dead,
embedding-NULL node.

**Decision (minimal, no migration):** on the dedup branch (`was_created==false`), fetch the row
via the existing `ClaimRepository::get_by_id` (returns a full `Claim` incl. `is_current`; reuse
its result for the §2.3 `trace_id` too). If `is_current == false`, **do not** silently return it —
respond **409 Conflict** with a clear message ("claim for (content_hash, agent_id) exists but is
not current — superseded/duplicate; refusing to attach ingestion to a tombstone"). Document the
**invariant: structural ingestion nodes (repo/PR/commit) are never superseded or marked-
duplicate** — a 409 here is an operational anomaly to investigate, not an expected path.

**Robust alternative (if structural nodes ever must support supersession):** replace
`uq_claims_content_hash_agent` with a **partial** `UNIQUE (content_hash, agent_id) WHERE
is_current` — this permits one live row alongside tombstones, so a live replacement can be
inserted. That **is a migration** (and changes dedup semantics), so it's out of this minimal
plan; flagged for the DID/structural-node work.

## 3. Fix B — agent-registration idempotency on `public_key`

### 3.1 Primary fix: a symmetric repo helper (recommended)
Add `AgentRepository::create_or_get(pool, agent) -> Result<(Agent, bool), DbError>` mirroring the
claim helper: `get_by_public_key` → existing; else `create`; on `DuplicateKey` (concurrent
registration) re-find; else `InvalidData`. **Document the constraint-match assumption** (as the
claim helper does, claim.rs:1901-1906): `create` maps *any* unique violation to `DuplicateKey`
without inspecting the constraint name (agent.rs:101-107); since `id` is a fresh UUID, the only
realistic collision is `public_key` — narrow this match if a future migration adds another unique
constraint on `agents`. All SQL stays in the repo layer (CLAUDE.md satisfied).

### 3.2 Wire into `create_agent` (`agents.rs:150`)
Replace `AgentRepository::create` (agents.rs:177) with `create_or_get`; return the existing agent
(real id) instead of 400. Guard **OAuth client provisioning** (agents.rs:181-210) with
`if was_created`. **Clarification (improvement):** `create_agent` only runs the scope check + OAuth
provisioning when `auth_ctx.is_some()`; the CLI `register_agent` POSTs with **no** bearer token, so
`auth_ctx` is `None` and provisioning never runs for it — the guard matters only for authenticated
(user/service-token) callers. `display_name`/labels divergence is ignored (first registration
wins; rename via `PUT /api/v1/agents/:id`).

### 3.3 Why this beats a new GET route
**Zero CLI change:** `register_agent`'s `resp.status().is_success()` (ingest_git.rs:939) now passes,
so the `exit(1)` paths never fire and the CLI gets the server id back via the normal `AgentResponse`.

### 3.4 Secondary / optional: `GET /agents?public_key=`
Defer (unnecessary once create is idempotent). **If added:** it is an agent-enumeration / key-
confirmation vector — it must inherit `list_agents`' auth posture and arguably gate behind a scope.
Reuses `get_by_public_key`, so no new `.sqlx`.

## 4. `content_hash` vs `(content_hash, agent_id)` — decision
Legacy `ClaimRepository::create` (claim.rs:120) / `create_with_tx` (claim.rs:300) dedup on
`content_hash` **alone** (claim.rs:136 / 317) — a documented noun-claim invariant violation
(claim.rs:106-115): agent B would get agent A's row, and the same commit content by different git
authors, or the same repo-node content under different orchestrators, would wrongly collapse.
**Decision: key on `(content_hash, agent_id)` via `create_or_get`** (matches the live constraint;
is what `POST /api/v1/claims` uses). **Do not** route `submit/packet` through legacy
`create`/`create_with_tx`. (Migrating the ~44 legacy callers is out of scope, claim.rs:113-115.)

**Note (improvement):** the `create_claim` template carries a comment (claims.rs:447-454) asserting
the constraint *was* dropped by "migration 107" and that `create_strict` "silently INSERTs
duplicates" — that is **internal-repo history, false in this public tree** (013 adds the constraint;
nothing drops it). Use `create_or_get` (the `if_not_exists=true` path); do not import the comment's
false assumption.

## 5. Embedding-policy safety on the return-existing path
Three embedding sites: the `cfg(feature="db")` arm at **submit.rs:1500**, its `cfg(not(db))` twin at
**~1551**, and the **figure-evidence** loop at **~1578**. Guard all three on
`was_duplicate == false` (mirror `claims.rs:590`) and use the **returned** id. The figure-evidence
loop iterates `evidence_ids` — on dedup those are **empty** (§2.3), so it is naturally inert, but
keep the `was_duplicate` guard too. The existing `is_host_telemetry` skip (submit.rs:1481-1498) is
preserved and composes. No cleanup-path (`supersede`/`mark_duplicate`) changes — this plan never
flips `is_current`.

## 7. Migrations & `cargo sqlx prepare`
- **No migration** for the core fix: `uq_claims_content_hash_agent` (013) and
  `agents_public_key_unique` (001) are already live, so the `DuplicateKey` race-catch path is live.
- **Likely no new `sqlx::query!` macro:** Fix A composes `create_or_get` (+ `get_by_id` for the §2.5
  guard / §2.3 trace_id) and the `properties` UPDATE that `create_claim` already uses; Fix B composes
  existing `get_by_public_key` + `create`. If implemented purely by reusing these, **no
  `cargo sqlx prepare`, no `.sqlx/` churn**. **Caveat:** if any step introduces a new/edited query
  macro (e.g. the §2.4 `properties` UPDATE is written inline in submit.rs rather than calling the
  existing repo method, or §2.5 needs a new is_current SELECT), run
  `DATABASE_URL=… cargo sqlx prepare --workspace -- --tests` and commit `.sqlx/`. Treat a
  `SQLX_OFFLINE` check failure as the signal.
- **Avoid `INSERT … ON CONFLICT … RETURNING`** in submit.rs — it would force a new macro + prepare;
  the reuse route is more minimal.
- **Prod constraint-liveness check (improvement):** §7's race-for-free argument assumes the
  constraints are live in the **prod DB** the GH-Actions ingester hits, not just the tree. Given the
  documented migration-drift hazards (and the misleading claims.rs:447 comment), confirm with a
  one-liner before relying on it: `\d claims` / `SELECT conname FROM pg_constraint WHERE conname IN
  ('uq_claims_content_hash_agent','agents_public_key_unique');`.

## 8. Durable `idempotency_key` — do NOT build a table
`(content_hash, agent_id)` + `create_or_get` **is** the durable cross-process key. Do not add an
idempotency-token table. Leave `state.idempotency_store` as a same-process fast-path (it short-
circuits before validation/persist and returns `was_duplicate:true` with a cached id — a different
latency/validation seam than the new DB path, which is fine; both legitimately set
`was_duplicate=true`). Document it as superseded for cross-run dedup; removal out of scope.

## 9. Plan #2 tweak — structural-node agent attribution (advisory)
The design doc (§6/§8) mandates find-or-create for repo/PR/commit nodes. Cross-run dedup keys on
`(content_hash, agent_id)`, so **every structural node must be attributed to a deterministic/stable
agent**:

- **Commit nodes** → deterministic **git-author** agent (Plan #2 §6.2/§6.3, `author_signer`) — stable. ✓
- **PR nodes** → the **orchestrator** agent resolved from the `Epigraph-Orchestrator-Id:` trailer /
  `EPIGRAPH_DEFAULT_ORCHESTRATOR_ID` (Plan #2 §6.1). This is stable **only if the orchestrator id is a
  real registered agent DID, not a per-run `Uuid::new_v4()`**. Plan #2's `run_pr_ingest` uses the
  trailer/env resolver (stable), so a re-run of the same PR dedups — **provided** the resolver never
  falls back to a random id. Make this explicit in Plan #2: the orchestrator id MUST be deterministic.
- **Repo-root node** → **must NOT use the per-PR orchestrator** (it varies across PRs, so the
  `agent_id` half differs and `create_or_get` cannot collapse the root → one duplicate root per PR).
  Attribute the repo-root to a **fixed, deterministic `git-ingester` system agent**. With Fix B, the
  first run find-or-creates that system agent once; every later run resolves the same id.

(The flat `ingest_git.rs` today uses `shared_agent_id = Uuid::new_v4()` at line 1545 — exactly the
anti-pattern to avoid for any shared structural node.)

## 10. Testing plan
DB: `DATABASE_URL=postgres://epigraph:epigraph@localhost/epigraph_db_repo_test`. Every new test gets
an adversarial-critic pass (reject tautology / happy-path-only / mock-shaped).

1. **Claim find-or-create across "runs."** Resubmit same `(content, agent_id)` from a fresh request →
   201, `was_duplicate==true`, **same `claim_id`**, and assert **row-count == 1** (not just 2xx).
2. **No double-embed on dedup.** Spy/mock the embedder (or assert the existing row's embedding /
   `updated_at` is unchanged) — dedup path issues no second embedding UPDATE.
3. **No evidence/trace 500 on dedup; empty evidence_ids returned (B2).** Resubmit a packet *with
   evidence* → no 500, no duplicate `evidence`/`reasoning_traces` rows, and the response
   `evidence_ids` is **empty** while `claim_id` is the canonical existing id.
4. **Distinct agents, same content → two claims.** Proves `(content_hash, agent_id)` keying, not
   `content_hash` alone.
5. **Agent create idempotent (Fix B).** Same public key twice → same agent id, success (not 400),
   exactly one `agents` row, OAuth provisioning at most once (authenticated caller).
6. **Concurrent race (optional).** Two simultaneous `create_or_get` for one key → one row via the
   `DuplicateKey` catch.
7. **Same-process resubmit WITHOUT idempotency_key (B6 regression, §6).** Assert it now returns
   `was_duplicate==true` + same id (documents the intended behavior change).
8. **Properties round-trip on the create path (B1).** Submit a packet with non-empty `properties`;
   assert the stored claim's `properties` matches (guards the data-loss regression).
9. **Dead-node guard (B3).** `mark_duplicate` a `(content_hash, agent_id)`, then resubmit it → assert
   **409** (not a silent tombstone return), per the never-supersede invariant.
10. **CI gate:** `cargo fmt --check`, `cargo clippy --workspace --locked -- -D warnings`, tests. If
    `SQLX_OFFLINE` check fails, an unintended new macro slipped in (§7) — investigate before commit.

## 11. Risks & open questions
1. **Pre-existing duplicate / superseded `(content_hash, agent_id)` rows on prod.** `find_by_…` uses
   `LIMIT 1` so it tolerates dups; **superseded** rows are handled by the §2.5 409 guard. A separate
   historical-dup/tombstone audit may still be warranted.
2. **AUTHORED-on-dedup semantics.** This plan does not re-emit AUTHORED on dedup (matches
   `create_claim`); MCP `create_claim_idempotent` does. If Plan #2 wants each PR-ingest recorded as an
   authorship event on the HTTP path, that's an explicit follow-up.
3. **`persist_packet` return-type ripple.** Contained (only `submit_packet` calls it), but the
   embedding/response/event/propagation/idempotency wiring (submit.rs ~1354-1671) and the
   `cfg(not(db))` twin must all consume the returned id (B4).
4. **Fixed-system-agent identity (Plan #2).** What keys the deterministic `git-ingester` agent
   (constant CI keypair vs UUIDv5 over a fixed namespace)? Defer to Plan #2 / the DID roadmap
   ([[project_epigraph_agent_identity]]); this plan only requires *some* stable id for the repo-root.
5. **Signatures stay disabled.** Both fixes assume `EPIGRAPH_REQUIRE_SIGNATURES` unset (design §6.3);
   enabling enforcement breaks the signer≠agent_id model independently of this plan.

## 12. Components (for the implementation plan)
1. `AgentRepository::create_or_get` (agent.rs) + constraint-match doc note.
2. `create_agent` wired to it, OAuth provisioning guarded on `was_created` (agents.rs).
3. `persist_packet`: `create_or_get` call; `was_created` short-circuit (evidence/trace/edges/
   propagation); `PersistOutcome` return; **properties UPDATE on create path**; **dead-node 409
   guard**; `cfg(not(db))` handling (submit.rs).
4. `submit_packet` caller rewiring: rebind `claim_id`, thread `was_duplicate`, guard all three
